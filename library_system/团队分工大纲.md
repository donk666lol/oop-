# 图书馆管理系统 - 5人分工大纲

## 项目概述
- **项目名称**: Library Management System (图书馆管理系统)
- **编程语言**: Python
- **小组规模**: 5人
- **开发周期**: 约8周

---

## 一、成员A - 系统架构与核心模块负责人

### 主要职责
1. **系统架构设计**
   - 设计整体系统架构
   - 定义模块间接口
   - 编写main.py入口程序

2. **核心模块开发**
   - `core/exceptions.py` - 自定义异常类
   - `core/library_system.py` - 主系统类
   - `core/data_manager.py` - 数据持久化

3. **项目管理**
   - 协调各成员进度
   - 代码审查与合并
   - 编写项目文档

### 技术要求展示
- 类与对象（系统架构）
- 封装（私有属性+属性访问器）
- 抽象类设计
- 自定义异常

### 交付物
- main.py
- core/ 目录所有文件
- README.md
- 项目进度报告

---

## 二、成员B - 数据结构模块负责人

### 主要职责
1. **线性数据结构**
   - `datastructures/doubly_linked_list.py` - 双向链表
   - `datastructures/stack.py` - 栈
   - `datastructures/queue.py` - 队列

2. **树形数据结构**
   - `datastructures/tree.py` - 通用树
   - `datastructures/binary_search_tree.py` - 二叉搜索树

3. **哈希结构**
   - `datastructures/hash_table.py` - 哈希表

### 技术要求展示
- 双向链表（必需）
- 栈（必需）
- 队列（必需）
- 树（必需）
- 二叉搜索树（必需）
- 哈希表/映射结构（必需）
- 递归树遍历（中序、前序、后序）

### 交付物
- datastructures/ 目录所有文件
- 数据结构性能分析文档
- 数据结构使用说明

## 协作流程

### 第1-2周：设计与框架
| 任务 | 负责人 | 交付物 |
|------|--------|--------|
| 系统架构设计 | A | 架构图、接口定义 |
| 数据结构框架 | B | 数据结构类框架 |
| 实体类设计 | C | 实体类初版 |
| 测试计划 | D | 测试计划文档 |
| 文档模板 | E | LaTeX模板 |

### 第3-4周：核心开发
| 任务 | 负责人 | 交付物 |
|------|--------|--------|
| 主系统实现 | A | library_system.py |
| 数据结构实现 | B | 所有数据结构 |
| 实体类完善 | C | 完整实体类 |
| UI框架 | D | 菜单系统 |
| 日志系统 | E | logger.py |

### 第5-6周：集成与测试
| 任务 | 负责人 | 交付物 |
|------|--------|--------|
| 模块集成 | A | 集成版系统 |
| 性能优化 | B | 优化后数据结构 |
| Bug修复 | C | 修复后的实体类 |
| 测试执行 | D | 测试报告 |
| 文档撰写 | E | 报告初稿 |

### 第7-8周：完善与提交
| 任务 | 负责人 | 交付物 |
|------|--------|--------|
| 最终审查 | A | 审查报告 |
| 代码整理 | B | 整理后代码 |
| 功能完善 | C | 完善后实体类 |
| 演示准备 | D | 演示脚本 |
| 最终提交 | E | 完整报告 |

## 注意事项

1. **代码规范**: 遵循PEP-8编码规范
2. **提交频率**: 每周至少提交一次进度
3. **代码审查**: 所有代码合并前需经A审查
4. **文档同步**: 代码更新后同步更新文档
5. **测试先行**: 开发功能前先编写测试用例
